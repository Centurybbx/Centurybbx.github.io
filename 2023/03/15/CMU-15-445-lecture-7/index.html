Hash Tables

Hash Table

哈希表的本质就是在一组无序的存有键值对的associative array。其空间复杂度为O(n)，时间复杂度平均为O(1)，最坏为O(n)。对于哈希表本身来说，其由两部分组成：

  • Hash Function
    • 如何在大的key空间上映射小的domain
    • 如何权衡快与慢（快的碰撞率高、满的碰撞率低）
  • Hashing Scheme
    • 如何处理key之间的哈希碰撞
    • 如何权衡直接申请大空间减少碰撞与使用额外策略解决碰撞

Hash Functions

哈希函数即对于其输入，输出一个整数形式表示的key。该函数是deterministic的，即对于相同的输入有共同的输出。（相反，如果对于相同的输入有着不同的输出，则称 此函数为probabilistic function）
在数据库领域使用的哈希函数与密码学领域有所不同，数据库领域的哈希函数主要在一两个指标：一是速度，二是碰撞率；而密码学的哈希函数会更侧重于不可反向推导（安全性）。
目前主流的哈希函数是Facebook XXHash3。

Static Hashing Schemes

静态哈希方案即哈希表的大小是固定的，当表被填满时需要重新申请一个新表（一般是原始大小的2倍），亦或是使用某种特定的措施。言而总之，当表满时会出现一定的问题，下面介绍几种静态哈希方案：

Linear Probe Hashing

最经典的通常也是最快的做法，即把buffer of array slots给抽象成一个环，对于可以直接插入的key-value则直接插入，而对于出现冲突的key就顺延，以此类推，就完成了插入操作，如下图。

对于这种哈希表的删除则需要注意，假如需要删除C|val这一行，删除后无法正确找到D对应的entry，因此对于删除我们可以做以下两种处理：

  • 在被删除的entry上加一个墓碑标志，表示其已经被删除，同时告知系统继续向下扫描
  • 把被删除的空余空间补齐

上面描述了一个key对应一个value的情形，但现实中还存在一种情况即一个key对应多个value，此时一般有两种解决办法：

  • 将一个key对应一组包含了多个value的列表
  • 把key-value一同作为entry存起来

Robin Hood Hashing

是线性探测方案的一个变式，其可以使得key的偏移总体均衡一些，不至于存在极端情况。思想是”劫富济贫”，对于偏移量大的，可使其替换偏移量小的，并将小的偏移量增加以达到平衡。

Cuckoo Hashing

使用多个哈希表来解决冲突。在不存在碰撞前直接插入即可，出现碰撞后倒推向前更改位置，并最终将提到的哈希重新计算后插入。（很奇怪的做法）

Dynamic Hashing Schemes

不同于静态哈希方案中的大小不可变，动态哈希方案中的哈希表随着需求动态变化（grow/shrink）。

Chained Hashing

最常见的动态哈希方案，在bucket满时可以动态增加bucket，很直观。

Extendible Hashing

对于上面提到了Chained Hashing方案，极端情况是所有key均碰撞，进而导致该bucket巨长，此时的哈希表与链表没什么区别。为了解决这种bucket很长的情况，Extendible Hashing出现了。

在Extendible Hashing中定义了几种属性：

  • Directories：含有指向Bucket的指针，大小为
  • Buckets：存哈希后的data，当local depth小于global depth时会存在多个指针同时指向一个bucket的情况
  • Global Depth：与directories有关，表示用来比较哈希后的key的比特数
  • Local Depth：与bucket有关，与Global depth定义类似，同时根据与global depth的大小比较来处理不同情况的overflow
  • Bucket Splitting：当处于bucket中的元素存满时，再次插入元素则需要split
  • Directory Expansion：当bucket overflow并且local depth等于global depth时会触发

其中关于Extendible Hashing如何扩容处理的细节不具体描述，等完成Project 1后再详细写。可参考：Extendible Hashing (Dynamic approach to DBMS)

Linear Hashing

相比于Extendible Hashing这种变化剧烈的方案，Linear Hashing偏向于”线性增长”。其策略是对于出现overflow的bucket，在directory中头部进行re-hash。之后出现此情况后以此类推。
