CMU 15-445 Tree Indexes

发表于2023-03-27|更新于2023-03-27|15445
|阅读量:

Trees Indexes

Table Indexes

在DBMS里面，存在有许多不同的数据结构来管理元数据、核心数据等，其中table index用来帮助提高查询效率。抽象地说，table index是一组包含了表中某些列的冗余数据，通过创建对应的索引可以辅助DBMS避免线性搜索，提高检索效率。
为了达到预期效果，table index中的内容应当是与表中的内容始终是同步的。通俗地讲，当表中数据发生变化时，索引中的数据应当进行相应的变化。
但对于开发人员来讲，当索引量增加时，数据检索快了，但同时增加了存储开销和管理开销，即在数据中创建索引也是存在trade-off的。

B+Tree

B+树是一种自适应的平衡树，其中的数据根据一定的规则有序放置，并且可以线性访问，插入和删除的时间复杂度也都保持在_O_(_log_(_n_))，因此它是十分适合应用于disk-oriented‘s DBMS。
对于B+树而言其存在两种形式的节点：

  • Inner Node（中间节点）
  • Leaf Node（叶子节点）

不同于B树，B+树将所有的数据存储在Leaf Node上，因此Inner Node常常被视作其下层节点的索引。（实现中，B+树会有不同的实现方式，即每个Leaf Node之间可能会有sibling pointers相连）

其中，Leaf Node的存值方式也有两种：

  1. Record IDs：记录一个存有value的tuple的指针
  2. Tuple data：即真正存放value的tuple

此外，B+树还有其他的特性：

  • 完美平衡（即所有的叶子节点都处于同一深度）
  • 每个中间节点（根节点除外）必须包含其最大长度的一半以上个元素
  • 每个带有n个元素的中间节点要有n+1个非空孩子

对于B+树的插入、删除、查询操作这里不详细展开叙述（因为Project 2中会具体实现，到时候再说它的坑等）。

Duplicate Keys

当B+树中存在重复key时，通常有两种做法：

  1. append record IDs：鉴于每个tuple的ID的独一无二的，所以将tuple加入到相同的key中融合成一个新的key，即可解决key重复的问题
  2. overflow nodes：对于重复的key，可以将其对应的Leaf Node上链接一个overflow node，并将重复的key放入新链接的overflow node即可

Clustered Indexes

基于B+树的结构和聚集索引的有序性特点，一个DBMS会默认采用聚集索引（默认主键，如果不指定则添加一个隐藏的主键）。当需要顺序扫描时（比如对于OLAP型数据库做数据分析时），clustered B+树的优势就出来了，它可以直接从左到右进行扫描处理。

Index Scan Page Sorting

对于采用了非聚集索引的表，如果直接对其进行顺序扫描，效率是非常低的。此时DBMS可以对于tuple中需要的数据进行排序，进而根据其page ID索取即可。

B+Tree Design Choices

Node Size

对于不同的需求，在B+树设计的过程中节点的大小选择也是不同的。通常来对，对于disk-oriented或者是读写速度较慢的存储介质，一般选择相对来说更大的节点，因为这样可以减少读取时间以提高效率；而对于在内存中的数据库，其可以使用相对较小的节点来存储以用来适配CPU cache中的page以填满整个page。

Merge Threshold

根据上面所提到过的B+树的几大特点，当数上发生了删除操作时可能会导致不平衡，进而需要merge的场景。但merge操作也存在一定的开销（写锁之类的），因此为了尽可能减少开销，会使用一些类似于“懒加载”的操作，也会在一定的情况下对需要merge的部分进行批处理，以减少开销。

Variable Length Keys

对于B+树中的节点，在key/value是定长的情况下，我们很容易合理设计其大小以满足填满page的需求。但如果key在可变长的情况下，怎么尽可能地减少浪费/降低开销是应当考虑的新问题，以下有几种解决措施：

  1. Pointers：为了避免直接存储变长的key，可选择使用指针表示key实际存在的地址，满足了定长节点包含变长的key。但这种情况需要对每个key搞一个指针，其唯一的应用场景是在嵌入式设备上
  2. Variable Length Nodes：直接将节点长度设置为可变，以达到key可变长的特性。但这种内存管理开销太大，一般不使用
  3. Padding：为了使可变长的key统一处理，直接将每个key的size设置为max size，不足的就padding。这种开销比上个还大，不可能使用
  4. Key Map/Indirection：使用索引来代替key-value

Intra-Node Search

B+树的结构使得从节点到节点之间的查询变得很快，但在节点内部进行查询仍然是一个需要考虑trade-off的事情：

  1. Linear：线性遍历节点中的所有key，这样做的好处是简单易行，无需对其进行排序，插入和删除也比较容易做；但坏处则是效率比较低，复杂度为_O_(_n_)
  2. Binary：由于节点内的keys是有序存放的，因此可以使用二分的方式进行查询，效率很高；但插入就相对更复杂一些，因为需要维护每个节点之间的排列
  3. Interpolation：根据已知的keys情况推断出当前需要插入的key的位置，属于特殊情况的操作

Optimizations

Pointer Swizzling

由于B+树中的节点是存在buffer pool中的page中，每次fetch时会有一定的开销。为了减少/避免这种开销，可以直接存对应page的指针，避免buffer pool的fetch操作。

Bucket Insert

在插入过程中如果满了则出现会需要分裂的情况，为了减少这种操作，已知B+树中Leaf Node可以通过sibling pointer相互连接，我们将一批需要插入的key-value组成一个简易的B+树，然后将这个树merge到原来的大树中。
